# i18n Design Spec

## Overview

Add internationalization (Chinese + English) to the food-tracker app. Default language is Chinese. Users can toggle to English via a UI switch. Language preference persists in localStorage.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| i18n library | None (custom Context + dictionary) | Only 2 languages, ~50 UI strings, static export — no need for next-intl/i18next overhead |
| English meal data | Separate file `data/meal-plan.en.md` | Clean separation, parser unchanged, each file independently readable |
| English MD headings | Keep Chinese headings | Headings are parser anchors only, never rendered — zero parser changes |
| Language toggle location | Top-right corner | Accessible without competing with bottom nav |
| Persistence | localStorage key `'locale'` | Consistent with existing shopping checklist pattern |
| Language switching | Pure client-side | Both datasets bundled at build time, instant toggle without page refresh |
| Routing | No route changes | Single set of routes, language is a client-side state |

## Data Layer

### Files

- `data/meal-plan.md` — Chinese (existing, unchanged)
- `data/meal-plan.en.md` — English (new, same structure with Chinese headings, English content)

**Important:** The English markdown must keep Chinese values in structured data fields that are used as type discriminators. Specifically, the 7-day execution table must use `虾仁日` / `牛肉日` in the day-type column, because `DayPlan.dayType` is typed as `'虾仁日' | '牛肉日'` and the parser/Zod schema enforce this. Translation of day-type labels happens at display time via the UI dictionary, not in the data.

### Changes to `lib/data.ts`

`loadMealPlan()` accepts a `locale` parameter:

- `'zh'` → reads `data/meal-plan.md`
- `'en'` → reads `data/meal-plan.en.md`

Caching keyed by locale (two separate cache slots).

New exports:

- `getMealPlan(locale: Locale): MealPlan`
- `getDayPlan(day: number, locale: Locale): DayPlan`
- `getShoppingList(locale: Locale): ShoppingList`

### Shopping item ID stability

Shopping item IDs are generated as `${category}:${name}` in the parser. Since English item names differ from Chinese, IDs will differ between locales. To keep checklist state stable across language switches, the English markdown must use the same item names as the Chinese version in the shopping list tables. The display names can be translated via a mapping in the UI layer, or alternatively, the parser can be updated to generate index-based IDs (e.g., `breakfast:0`, `breakfast:1`). **Recommended: use index-based IDs** — this is a small parser change that eliminates the coupling entirely.

### No changes to

- `lib/parser.ts` — English MD uses Chinese headings, parsing logic unchanged (except optional index-based ID change)
- `lib/schema.ts` — Zod schemas validate structure, not language

### Server → Client data flow

```
app/page.tsx (server component)
  getMealPlan('zh') + getMealPlan('en')
  → pass BOTH full MealPlan objects as props to HomepageClient

app/shopping/page.tsx (server component)
  getShoppingList('zh') + getShoppingList('en')
  → pass both as props to ShoppingChecklist
```

Client components use `useLocale()` to select which dataset to render.

### Page restructuring: `app/page.tsx`

Currently, `DayContent` components are rendered directly in the server component `HomePage`, outside of `HomepageClient`. Since `DayContent` needs to render locale-specific data, the page must be restructured:

- `app/page.tsx` (server) loads both `getMealPlan('zh')` and `getMealPlan('en')`, passes both to `HomepageClient`
- `HomepageClient` (client) takes over rendering of `DayContent` components, selecting the correct locale's data via `useLocale()`
- `DayContent`, `WeighingCard`, `BreakfastSection` become client components (or receive translated strings as props from `HomepageClient`)

This is the most significant architectural change. The current pattern of server-rendered `DayContent` blocks with client-side `data-day` show/hide toggling will be replaced by client-side rendering of the active day's content with locale-aware data selection.

## i18n Infrastructure

### New file: `lib/i18n.tsx`

```ts
type Locale = 'zh' | 'en';

// LanguageContext: stores locale + toggleLocale()
// LanguageProvider: reads localStorage('locale'), defaults to 'zh'
//   - Uses useEffect to read localStorage (avoids SSR hydration mismatch)
//   - Server-side and initial render always use 'zh' (default)
//   - After mount, if localStorage has 'en', triggers re-render
// useLocale(): returns current Locale
// useT(): returns t(key) function that looks up translations
```

`LanguageProvider` wraps the app in `app/layout.tsx` via a client component wrapper.

### SSR hydration considerations

- `<html lang="zh-CN">` is set statically in the server component `layout.tsx`. A client-side `useEffect` in `LanguageProvider` updates `document.documentElement.lang` to `'en'` when needed. This avoids hydration mismatch (initial render always matches server: `zh-CN`).
- `metadata` (title/description) is a static server-side export and cannot respond to client-side state. The static HTML will always have the Chinese title. A client-side `useEffect` updates `document.title` when locale changes.

### New file: `lib/translations.ts`

Flat `Record<string, { zh: string; en: string }>` containing all ~50 UI strings:

- Navigation: `nav.today`, `nav.shopping`
- Day types: `day.shrimp`, `day.beef`
- Weekdays: `weekday.sun` through `weekday.sat`
- Weighing labels: `weigh.you`, `weigh.wife`, `weigh.rice`, `weigh.vegetable`, `weigh.oil`, `weigh.protein_powder`, `weigh.cooked_weight`, `weigh.raw_weight`
- Section titles: `section.breakfast_details`, `section.recipe_steps`, `section.prep_reminders`, `section.meal_prep_tips`, `section.execution_reminders`
- Shopping: `shopping.title`, `shopping.reset`, `shopping.confirm_reset`, `shopping.progress`
- Shopping categories: `shopping.cat.breakfast`, `shopping.cat.staple`, `shopping.cat.protein`, `shopping.cat.vegetable`, `shopping.cat.oil`, `shopping.cat.pantry`
- Breakfast: `breakfast.smoothie_title`, `breakfast.you`, `breakfast.wife`
- Day navigation: `daynav.prev`, `daynav.next`, `daynav.jump_today`, `daynav.today`
- Date format: `date.format` (zh: `${month}月${date}日`, en: `MMM D`)
- Misc: `weigh.subtitle`

## Language Toggle UI

### New file: `components/LanguageToggle.tsx`

- Capsule-shaped button in top-right corner, fixed position
- Displays "中 / EN" with current language highlighted
- Calls `toggleLocale()` on click
- Warm color palette consistent with existing design

## Component Changes

All components with hardcoded Chinese strings need `useT()` calls. Since `useT()` is a hook, these components must be client components (`'use client'`).

Components that are currently server components and need conversion:

- `DayContent.tsx` → client component
- `WeighingCard.tsx` → client component
- `BreakfastSection.tsx` → client component

Components that are already client components:

- `DayNavigation.tsx` — already `'use client'`
- `BottomNav.tsx` — already `'use client'` (if not, convert)
- `ShoppingChecklist.tsx` — already `'use client'`

| Component | Strings to replace |
|-----------|-------------------|
| `BottomNav.tsx` | nav labels (2) |
| `DayNavigation.tsx` | weekday names, day types, aria-labels, today indicator, date format (8+) |
| `WeighingCard.tsx` | person labels, food labels, weight annotations, subtitle (8+) |
| `BreakfastSection.tsx` | section title, person labels (3) |
| `DayContent.tsx` | section titles, smoothie heading (5+) |
| `ShoppingChecklist.tsx` | title, categories, progress text, reset button, confirm dialog (12+) |

### Components NOT changing

- `CollapsibleSection.tsx` — receives `title` as a prop, no hardcoded strings
- `ShoppingCategory.tsx` — receives translated props from `ShoppingChecklist`
- `ShoppingItemRow.tsx` — displays data from `ShoppingItem` type
- `WKWebViewScrollFix.tsx` — language-independent

### Data-switching in client components

- `HomepageClient`: receives `mealPlanZh` + `mealPlanEn` props, selects by `useLocale()`. Now also responsible for rendering `DayContent` components (moved from `app/page.tsx`).
- `ShoppingChecklist`: receives `shoppingListZh` + `shoppingListEn` props, selects by `useLocale()`

### `lib/date-utils.ts`

- `getDayLabel()` accepts a `locale` parameter
- `getCalendarDate()` in `DayNavigation.tsx` formats dates based on locale: zh → `3月18日`, en → `Mar 18`

## What is NOT changing

- `lib/parser.ts` — no changes (except optional index-based shopping item IDs)
- `lib/schema.ts` — no changes
- `lib/shopping-storage.ts` — localStorage keys unchanged, language-independent
- `lib/types.ts` — no changes (dayType remains Chinese string union)
- Routing structure — no new routes
- Build configuration — no changes
- `WKWebViewScrollFix` — language-independent
