# i18n Design Spec

## Overview

Add internationalization (Chinese + English) to the food-tracker app. Default language is Chinese. Users can toggle to English via a UI switch. Language preference persists in localStorage.

This is a personal-use app; SEO and shareable metadata are not goals.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| i18n library | None (custom Context + dictionary) | Only 2 languages, ~50 UI strings, static export — no need for next-intl/i18next overhead |
| English meal data | Separate file `data/meal-plan.en.md` | Clean separation, each file independently readable |
| English MD anchors | Keep all Chinese parser anchors in English MD | Headings, subsection labels, and sentinel values are parser anchors only, never rendered — minimizes parser changes |
| Language toggle location | Top-right corner | Accessible without competing with bottom nav |
| Persistence | localStorage key `'locale'` | Consistent with existing shopping checklist pattern |
| Language switching | Pure client-side | Both datasets bundled at build time, instant toggle without page refresh |
| Routing | No route changes | Single set of routes, language is a client-side state |

## Data Layer

### Files

- `data/meal-plan.md` — Chinese (existing, unchanged)
- `data/meal-plan.en.md` — English (new, same structure, English content)

**Required Chinese anchors in English MD:** The English markdown must preserve the following Chinese strings exactly, because the parser uses them as section/subsection locators or sentinel values:

- **Section headings**: `固定早餐`, `7 天执行表`, `每天实际称重清单`, `一周只买一次菜：购物清单`, `每道菜超简做法`, `备餐建议`, `最后执行提醒`
- **Subsection labels**: `奶昔总配方`, `早餐部分`, `主食部分`, `蛋白质部分`, `午晚餐蔬菜部分`, `建议买菜分配`, `食用油与调味料`
- **Person identifiers in tables**: `你`, `你老婆`
- **Day type values**: `虾仁日`, `牛肉日` (typed as `'虾仁日' | '牛肉日'` in `DayPlan.dayType`)
- **Quantity sentinel**: `适量` (used to classify pantry vs. oil items and to produce null quantities)

All other content (ingredient names, recipe steps, tips, reminders, shopping item names, breakfast descriptions) should be in English.

### Required parser changes (`lib/parser.ts`)

Despite keeping Chinese anchors, two parser changes are needed:

1. **Breakfast whitespace**: `parseFixedBreakfast()` calls `content.replace(/\s+/g, '')` on breakfast table cells. This collapses Chinese text harmlessly but destroys English text (e.g., `"2 veggie buns + soy milk"` → `"2veggiebuns+soymilk"`). **Fix**: remove the whitespace stripping (`.trim()` is sufficient).

2. **Shopping item IDs**: Currently `${category}:${name}`. Since English item names differ from Chinese, IDs must be locale-independent. **Fix**: use category-scoped index IDs (`${category}:${index}`). Bump the storage key version from `v1` to `v2` in `lib/shopping-storage.ts` so old saved state is cleanly discarded.

### Changes to `lib/data.ts`

`loadMealPlan()` accepts a `locale` parameter:

- `'zh'` → reads `data/meal-plan.md`
- `'en'` → reads `data/meal-plan.en.md`

Caching keyed by locale (two separate cache slots).

New exports:

- `getMealPlan(locale: Locale): MealPlan`
- `getDayPlan(day: number, locale: Locale): DayPlan`
- `getShoppingList(locale: Locale): ShoppingList`

### Server → Client data flow

```
app/page.tsx (server component)
  getMealPlan('zh') + getMealPlan('en')
  → pass BOTH full MealPlan objects as props to HomepageClient

app/shopping/page.tsx (server component)
  getShoppingList('zh') + getShoppingList('en')
  → pass both as props to ShoppingChecklist
```

Both datasets are small (~7-day meal plan, <30KB per locale). Duplicating them in props is acceptable for this app size and avoids lazy-loading complexity.

Client components use `useLocale()` to select which dataset to render.

### Page restructuring: `app/page.tsx`

Currently, `DayContent` components are rendered directly in the server component `HomePage`, outside of `HomepageClient`. Since `DayContent` needs to render locale-specific data, the page must be restructured:

- `app/page.tsx` (server) loads both `getMealPlan('zh')` and `getMealPlan('en')`, passes both to `HomepageClient`
- `HomepageClient` (client) takes over rendering of `DayContent` components, selecting the correct locale's data via `useLocale()`
- `DayContent`, `WeighingCard`, `BreakfastSection` become client components (they need `useT()` hooks for their hardcoded strings)

This moves rendering from server to client. The tradeoff is acceptable: this is a static export served from CDN with no SSR benefit, and the app is personal-use with no SEO requirements. The no-JS fallback is not a concern.

The following must be removed:

- `DayContent.tsx`: the `data-day` attribute and `className="hidden"` on the wrapper div
- `DayNavigation.tsx`: the `showDay()` function and its `useEffect` that does `document.querySelectorAll('[data-day]')` DOM manipulation
- `app/page.tsx`: the `.map()` that renders all 7 `DayContent` blocks — this moves into `HomepageClient`

## i18n Infrastructure

### `Locale` type

The `Locale` type (`'zh' | 'en'`) goes in `lib/types.ts`, not in the provider module. This allows both server-side code (`lib/data.ts`) and client-side code (`lib/i18n.tsx`) to import it without cross-boundary dependencies.

### New file: `lib/i18n.tsx`

```ts
// LanguageContext: stores locale + toggleLocale()
// LanguageProvider: reads localStorage('locale'), defaults to 'zh'
//   - Uses useEffect to read localStorage (avoids SSR hydration mismatch)
//   - Server-side and initial render always use 'zh' (default)
//   - After mount, if localStorage has 'en', triggers re-render
// useLocale(): returns current Locale
// useT(): returns t(key: TranslationKey) function that looks up translations
```

`LanguageProvider` wraps the app in `app/layout.tsx` via a client component wrapper.

### Hydration and flash mitigation

- `<html lang="zh-CN">` is set statically in the server component `layout.tsx`. A client-side `useEffect` in `LanguageProvider` updates `document.documentElement.lang` to `'en'` when needed.
- To minimize the Chinese-first flash for returning English users: inject a small inline `<script>` in `layout.tsx` that reads `localStorage('locale')` and sets a `data-locale` attribute on `<html>` before React hydrates. The `LanguageProvider` reads this attribute for its initial state instead of defaulting to `'zh'`, avoiding a re-render flash.
- `metadata` (title/description) is a static server-side export. The static HTML will always have the Chinese title/description. A client-side `useEffect` updates `document.title` when locale changes. This is acceptable since SEO/share metadata is out of scope for this personal-use app.

### New file: `lib/translations.ts`

Typed dictionary using `as const` for compile-time key safety:

```ts
export const translations = {
  'nav.today': { zh: '今日', en: 'Today' },
  'nav.shopping': { zh: '采购清单', en: 'Shopping List' },
  // ... ~50 keys
} as const;

export type TranslationKey = keyof typeof translations;
```

The `useT()` hook returns `t(key: TranslationKey)`, so typos and missing keys are caught at compile time.

Key groups:

- Navigation: `nav.today`, `nav.shopping`
- Day types: `day.shrimp`, `day.beef`
- Weekdays: `weekday.sun` through `weekday.sat`
- Weighing labels: `weigh.you`, `weigh.wife`, `weigh.rice`, `weigh.vegetable`, `weigh.oil`, `weigh.protein_powder`, `weigh.cooked_weight`, `weigh.raw_weight`
- Section titles: `section.breakfast_details`, `section.recipe_steps`, `section.prep_reminders`, `section.meal_prep_tips`, `section.execution_reminders`
- Shopping: `shopping.title`, `shopping.reset`, `shopping.confirm_reset`, `shopping.progress`
- Shopping categories: `shopping.cat.breakfast`, `shopping.cat.staple`, `shopping.cat.protein`, `shopping.cat.vegetable`, `shopping.cat.oil`, `shopping.cat.pantry`
- Breakfast: `breakfast.smoothie_title`, `breakfast.you`, `breakfast.wife`
- Day navigation: `daynav.prev`, `daynav.next`, `daynav.jump_today`, `daynav.today`
- Date format: `date.format` (zh: `${month}月${date}日`, en: `MMM D`)
- Misc: `weigh.subtitle`

## Language Toggle UI

### New file: `components/LanguageToggle.tsx`

- Capsule-shaped button in top-right corner, fixed position
- Displays "中 / EN" with current language highlighted
- Calls `toggleLocale()` on click
- Warm color palette consistent with existing design
- Placement: `z-index` above page content but below modals, `safe-area-inset` padding for iOS, minimum 44×44px touch target, positioned to avoid overlapping the day navigation card

## Component Changes

All components with hardcoded Chinese strings need `useT()` calls. Since `useT()` is a hook, these components must be client components (`'use client'`).

Components that are currently server components and need conversion:

- `DayContent.tsx` → client component
- `WeighingCard.tsx` → client component
- `BreakfastSection.tsx` → client component

Components that are already client components:

- `DayNavigation.tsx` — already `'use client'`
- `BottomNav.tsx` — already `'use client'`
- `ShoppingChecklist.tsx` — already `'use client'`

| Component | Strings to replace |
|-----------|-------------------|
| `BottomNav.tsx` | nav labels (2) |
| `DayNavigation.tsx` | weekday names, day types, aria-labels, today indicator, date format (8+) |
| `WeighingCard.tsx` | person labels, food labels, weight annotations, subtitle, protein emoji logic (8+) |
| `BreakfastSection.tsx` | section title, person labels (3) |
| `DayContent.tsx` | section titles, smoothie heading (5+) |
| `ShoppingChecklist.tsx` | title, categories, progress text, reset button, confirm dialog (12+) |

### `WeighingCard.tsx` protein emoji fix

The `proteinEmoji()` function checks `name.includes('虾')` to choose between 🦐 and 🥩. This breaks with English protein names. **Fix**: derive the emoji from `dayType` (passed as a new prop) instead of from the protein name. `虾仁日` → 🦐, `牛肉日` → 🥩.

### Components NOT changing

- `CollapsibleSection.tsx` — receives `title` as a prop, no hardcoded strings
- `ShoppingCategory.tsx` — receives translated props from `ShoppingChecklist`
- `ShoppingItemRow.tsx` — displays data from `ShoppingItem` type
- `WKWebViewScrollFix.tsx` — language-independent

### Data-switching in client components

- `HomepageClient`: receives `mealPlanZh` + `mealPlanEn` props, selects by `useLocale()`. Now also responsible for rendering `DayContent` components (moved from `app/page.tsx`).
- `ShoppingChecklist`: receives `shoppingListZh` + `shoppingListEn` props, selects by `useLocale()`

### `lib/date-utils.ts`

- `getDayLabel()` accepts a `locale` parameter
- `WEEKDAY_NAMES` array replaced by translation dictionary lookups (`weekday.sun` through `weekday.sat`)
- `getCalendarDate()` in `DayNavigation.tsx` formats dates based on locale: zh → `3月18日`, en → `Mar 18`
